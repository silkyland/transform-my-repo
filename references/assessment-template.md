# Assessment Document Template

Write to `docs/TRANSFORM.md` (or the user's chosen path). Every section
required, in order. Tag claims per the family rules; source claims carry
`file:line`, target-ecosystem claims carry URL + version.

---

```markdown
# Transformation Assessment: <source> → <target>

> Generated by transform-my-repo on <date>. Driver: <the WHY, one line>.

## 1. Verdict

**GO / PARTIAL / NO-GO** — <two-line justification tied to the driver>.

<For PARTIAL: exactly what migrates and what stays/bridges.
For NO-GO: the recommended alternative and its evidence.>

## 2. Scope and constraints

Source, target (versions), transformation type(s) per the catalog,
constraints, non-goals. If combined types: the chosen sequencing and why.

## 3. Dependency census & target coverage

| Dependency | Role | Call sites | Target equivalent | Coverage | Evidence |
|------------|------|-----------:|-------------------|----------|----------|

**Missing equivalents (headline):** each with its options and chosen handling.

## 4. Platform couplings and semantic gaps

| Gap | Where it bites | Handling | Evidence |
|-----|----------------|----------|----------|
| <runtime/typing/numeric/...> | `file:line` | <test / redesign / bridge> | <target docs URL> |

## 5. Difficulty heatmap

Ranked hardest-first, per the rubric. Worst factor named per row.

| Module | Size | Deps | Coupling | Tests | Churn | Total | Worst factor |

**What will be hard:** <prose summary of the top rows — the honest answer.>

## 6. Parity harness plan

What pins current behavior before anything moves: which seams
(HTTP/DB/files), which modules need characterization tests written first,
estimated share of total effort. Modules with no seam get one built —
listed explicitly.

## 7. Migration strategy

- **Pattern:** <strangler fig / branch-by-abstraction / module-by-module / big-bang(justified)> — decided, not a menu.
- **Coexistence bridge:** <how old and new run together; data consistency ownership>.
- **Cutover & rollback:** per-phase triggers and procedures.

## 8. Phased roadmap

Phase 1 is a low-risk representative module — the estimate calibrator.
Each phase sized for one deep-plan run:

### Phase <N>: <module group> (magnitude: S/M/L/XL)
- **Scope:** concrete modules/files
- **Parity gate:** <the tests/diff that must be green to call it done>
- **Rollback point:** <how to back out>
- **Next:** run deep-plan on this phase for the implementation plan

## 9. Risk register

| Risk | Likelihood | Impact | Mitigation / rollback |

## 10. Corrections & unverified

Beliefs corrected during analysis (mandatory — an assessment that never
found itself wrong didn't check), and anything still UNVERIFIED with what
would verify it (e.g. "target perf claim — needs the Phase 1 spike").
```

---

## Quality bar

- The verdict section alone convinces a skeptical CTO, because every number
  in it is traceable to sections 3–5.
- Phase 1 can be handed to deep-plan today.
- The "what will be hard" prose matches the heatmap — no smoothing.
